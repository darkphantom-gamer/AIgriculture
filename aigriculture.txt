--------------------------------------------------------------------------------
1. WHAT IT IS
--------------------------------------------------------------------------------
AIgriculture is a self-hosted smart-farm controller for a small strawberry farm.
One Raspberry Pi runs the whole stack and exposes a web dashboard. It does five
things at once:

  (1) IRRIGATION   — reads soil moisture from 8 plants and drives 8 water pumps
                     with a safe burst-watering cycle.
  (2) FARMMONITOR  — a camera periodically scans the plants and classifies
                     DISEASE and fruit RIPENESS using vision models; results are
                     stored and emailed.
  (3) SECURITY CAM — a second camera watches for intruders (people/animals) and
                     raises an on-site siren + dashboard alert.
  (4) FLORA        — an AI farm assistant you can chat with (web or LoRa radio)
                     that can read live sensor state and control the farm.
  (5) DASHBOARD    — a FastAPI web app (login, live video, charts, history,
                     analytics, settings) reachable at http://<pi-ip>:8000.

DESIGN RULE FOR THE PUBLIC VERSION (important):
  The DEFAULT / PRIMARY path must run on a PLAIN Raspberry Pi with NO special
  hardware accelerator — vision runs on the CPU with a small YOLO model.
  The Hailo-10H AI HAT is an OPTIONAL, secondary backend for users who own one.
  Everything (dashboard, irrigation, security, FarmMonitor, FLORA) must work
  without Hailo. See section 6.

2. HARDWARE — BILL OF MATERIALS
--------------------------------------------------------------------------------
  · Raspberry Pi 5 (4 GB+ ; 8 GB recommended)            ×1
  · MicroSD card (32 GB+) or NVMe via PCIe                ×1
  · ADS1115 16-bit I2C ADC boards                         ×2  (4 channels each = 8 sensors)
  · Capacitive soil-moisture sensors (analog)             ×8
  · 8-channel relay board (active-LOW, opto-isolated)     ×1
  · Small DC water pumps (one per plant)                  ×8  (driven through relays)
  · Cameras (pick what you have — see section 5)          ×2  (one FarmMonitor, one Security)
  · Passive piezo buzzers                                 ×2  (intruder siren)
  · External 5V/12V supply for pumps (NOT from Pi rail)   ×1
  · OPTIONAL: Hailo-10H AI HAT (M.2) for accelerated vision
  · OPTIONAL: Meshtastic LoRa radio (e.g. Heltec/LilyGo) for off-grid chat

--------------------------------------------------------------------------------
3. WIRING — EXACT PIN MAP  (BCM numbering)
--------------------------------------------------------------------------------
These are the pins the code uses. They live in ONE place in the source and are
meant to be edited there if your wiring differs (see aigriculture/hardware/gpio.py
and aigriculture/hardware/moisture.py).

A) SOIL MOISTURE  — via I2C (ADS1115 ×2)
   Pi I2C bus 1:  SDA = BCM 2 (phys pin 3),  SCL = BCM 3 (phys pin 5)
   ADS1115 #1 @ I2C 0x48  → A0,A1,A2,A3  → plants A,B,C,D
   ADS1115 #2 @ I2C 0x49  → A0,A1,A2,A3  → plants E,F,G,H
   (8 capacitive sensors total. Calibrate dry/wet raw values per sensor.)

B) WATER PUMPS  — via 8-channel relay board (ACTIVE-LOW: LOW = pump ON)
   Plant A → BCM 17        Plant E → BCM 5
   Plant B → BCM 27        Plant F → BCM 6
   Plant C → BCM 22        Plant G → BCM 13
   Plant D → BCM 23        Plant H → BCM 19
   Pumps are powered from an EXTERNAL supply; relays only switch them.

C) INTRUDER SIREN  — 2 passive piezo buzzers (PWM tone, 2700 Hz)
   Buzzer 1 → BCM 18 (PWM-capable)
   Buzzer 2 → BCM 12 (PWM-capable)
   Both fire together; triggered ONLY by the security camera threat hook.

D) RESERVED / DO-NOT-REUSE
   I2C uses BCM 2,3. Relays use BCM 5,6,13,17,19,22,23,27. Buzzers use 12,18.
   If you add hardware, avoid those pins.

E) CAMERAS
   FarmMonitor camera and Security camera are independent video sources.
   They can be Pi CSI (ribbon), USB (/dev/videoN), or network (RTSP/HTTP IP).
   Selected at launch with --camera (see section 5).

--------------------------------------------------------------------------------
4. SUBSYSTEMS — WHAT EACH ONE DOES
--------------------------------------------------------------------------------
4.1 IRRIGATION ENGINE
   · Polls all active plants' moisture on a loop.
   · BURST CYCLE: pump ON 3 s, then SOAK 10 s, re-measure, repeat until target.
     (Burst+soak avoids over-watering and channeling in the soil.)
   · 70% MOISTURE SAFETY HARDLOCK: a pump is never energized if that plant is
     already at/above 70% moisture, regardless of any other request. This is a
     hard safety stop, not a preference.
   · 8 plants exist in hardware (A–H); a runtime registry (.plants.json) picks
     which are ACTIVE (default: A,B). Enable/disable without restart.
   · Manual + automatic watering; all actions logged.

4.2 FARMMONITOR (disease + ripeness)
   · On a schedule (default hourly) the FarmMonitor camera captures a BATCH of
     frames (quality-gated: rejects blurry / too-dark / too-bright frames).
   · Two models run over the batch:
       DISEASE model   → 5 classes: Anthracnose Fruit Rot, Powdery Mildew (Fruit),
                         Powdery Mildew (Leaf), Gray Mold, Angular Leafspot.
       RIPENESS model  → 5 classes: Flowering, Green, Turning, Nearly Ripe, Ripe.
   · MULTI-CYCLE scan: several capture/scan cycles are fused into one session for
     a stable verdict (reduces single-frame false positives).
   · Results (labels, confidence, annotated images) are saved to storage and can
     trigger an EMAIL alert with images.

4.3 SECURITY CAMERA
   · Continuously runs object detection on the security feed.
   · Only cares about a small set of "threat" classes (e.g. person, and large
     animals) — everything else is ignored to keep CPU load low.
   · On a confirmed threat: fires the dual buzzer siren, logs a storage event
     with a snapshot, and pushes a live dashboard alert.
   · Siren has a master mute (admin or FLORA can toggle).

4.4 FLORA — AI FARM ASSISTANT  (3 layers)
   LAYER 1 (Chat):   Multi-provider LLM chat (Groq / Cerebras / Mistral / Gemini),
                     with automatic key round-robin and a FULL OFFLINE fallback
                     (rule-based replies) when no API keys are set or quota is hit.
   LAYER 2 (Tools):  FLORA can call real farm tools that read/write LIVE state:
                     get moisture, get camera status, run a FarmMonitor scan now,
                     water a plant, list/analyze storage events, compile a report,
                     send an email. Tools operate on the same in-memory state the
                     dashboard uses.
   LAYER 3 (Auto):   Natural-language task scheduler (APScheduler) — e.g.
                     "scan the plants every morning at 7" — plus a 48-hour PDF
                     farm report builder (moisture trend chart + findings).
   FLORA chat is reachable from the dashboard (WebSocket, JWT-auth, with an HTTP
   fallback) and from a Meshtastic LoRa radio (see 4.6).

4.5 DASHBOARD (web app)
   · FastAPI + WebSocket backend; single-page HTML front-end.
   · Login (JWT cookie/token), role-aware (admin vs viewer).
   · Live MJPEG video for both cameras, live moisture gauges, irrigation
     controls, FarmMonitor results gallery, security event log, analytics
     charts, FLORA chat, and a settings page.
   · Static images are served through opaque hash-only URLs (you cannot guess a
     filename to scrape stored photos).

4.6 MESHTASTIC LoRa BRIDGE
   · Connects to a local meshtasticd daemon over TCP.
   · GENERAL RULE: whatever channel or direct message a text arrives on, FLORA
     processes it and replies ONLY back to that same channel/DM. No broadcasting.
   · Long replies are trimmed/segmented to fit LoRa's small payload.
   · Optional allow-list of node IDs (config-driven, blank = allow all).

4.7 EMAIL / ALERTS
   · SMTP (any provider) for: FarmMonitor disease/ripeness alerts (with images),
     security alerts, and FLORA's PDF reports.
   · Recipient + SMTP creds come from config.yaml (never hard-coded).

4.8 STORAGE / DATABASE
   · SQLite for users, auth tokens, and settings.
   · Event store on disk: each security/farm event = a JSON record
     (time, event_type, label, confidence, images[]) plus image files.
   · A retention/cleanup routine prunes old scans.

5. CAMERA SELECTION (user-chosen at launch)
--------------------------------------------------------------------------------
The user picks each camera with a --camera argument. Support three source types:

  · Pi CSI camera (libcamera/picamera2):   --camera csi:0   (and csi:1)
  · USB UVC camera (OpenCV/V4L2):          --camera /dev/video0
  · Network IP camera (RTSP or HTTP):      --camera rtsp://user:pass@192.168.1.50/stream

Two cameras are used: one for FarmMonitor, one for Security. Example:
  python -m aigriculture --farm-camera /dev/video0 --security-camera /dev/video2
The system must degrade gracefully: if a camera is missing, that feature is
disabled but the rest of the dashboard still runs.

--------------------------------------------------------------------------------
6. VISION BACKENDS — NON-HAILO IS PRIMARY
--------------------------------------------------------------------------------
There is ONE detector interface with two implementations behind it:

  DEFAULT  → CPU YOLO (Ultralytics, e.g. yolo11n / yolov8n). Pure pip install,
             runs on any Pi. For the security camera, keep it fast by:
               - processing every Nth frame (frame-skip),
               - filtering to only the threat classes you care about,
               - using a small input size (e.g. 320–480 px).
  OPTIONAL → Hailo-10H (.hef models) via the Hailo runtime + GStreamer, for users
             who own the AI HAT and want high FPS. Enabled with --backend hailo.

The FarmMonitor disease/ripeness models ship as PyTorch .pt (CPU) by default,
with an optional pre-compiled .hef for the Hailo path. Model files live in
models/ (see models/README.md for where to download / how to retrain).

--------------------------------------------------------------------------------
8. HOW TO RUN
--------------------------------------------------------------------------------
A) Docker (recommended, non-Hailo):
     git clone <repo> && cd AIgriculture
     cp .env.example .env && cp config.example.yaml config.yaml   # edit them
     docker-compose up -d
     open http://<pi-ip>:8000
B) Native (Python 3.11+):
     python -m venv .venv && source .venv/bin/activate
     pip install -r requirements.txt
     python -m aigriculture --farm-camera /dev/video0 --security-camera /dev/video2
C) Hailo (optional add-on):
     pip install -r requirements-hailo.txt
     docker-compose -f docker-compose.yml -f docker-compose.hailo.yml up -d
     # or native:  python -m aigriculture --backend hailo ...
